# 多结局系统重设计

## 现状分析

**数值曲线：**
- 仅计算主线必过事件（不含任何分支选项），`anya_affection` 已达 300+，`trust_level` 已达 200+
- 完美结局阈值：`affection≥150 AND trust≥125` → **必然达到**
- 普通结局阈值：`affection≥95 AND trust≥70` → **必达**
- 三人称"bad ending"实际只存在于第一章选选项3

**分支空洞：**
- `branch_type` 只有两处使用（L1112 停电小对话、L1240 萤火虫牵手），影响极小
- 第4章"我先回去了"仍 `jump hot_pot_date` → 殊途同归
- 第5章告白分支、第6章结局分支**实际上永远进完美**，因为永远达到阈值

## 重设计方案

### 1. 提高结局阈值 + 新增te

```
完美结局：affection ≥ 250 AND trust ≥ 200 AND branch_type == 0
普通结局：affection ≥ 150 AND trust ≥ 100 AND branch_type != 2
搭讪结局：affection ≥ 180 AND trust ≥ 120 AND branch_type == 1  (新增)
遗憾结局：affection ≥ 80  AND trust ≥ 50
坏结局：   低于以上或 branch_type == 2
```

### 2. 全面削减数值

将主线的 **自动好感奖励减半**，让分支选择成为关键。

### 3. 增加真正有后果的分支

- **第4章菜单**：选"先回去" → 好感不增加+错过火锅约会 → jump 到被削弱的路线
- **新增几个扣分选项**：在每个菜单中给一个减好感/信任的选项
- **让 branch_type 在更多地方产生分支**

### 4. 第6章结局

- **完美结局**（主线+branch_type=0+高值）：拒绝收购，品牌独立，告白成功，南山看星星
- **普通结局**（高值但非完美条件）：接受收购，成为恋人但异地
- **搭讪结局**（branch_type=1+高值）：不同风格的故事走向
- **遗憾结局**（低值）：收购导致离别

### 5. 配置文件

新增一个 `ending_paths.rpy` 来存放结局标签和条件判断（避免 script.rpy 过长）。
